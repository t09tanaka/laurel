# AGENTS.md

## 言語設定

- 応答は常に日本語で行う。
- 内部の技術判断は英語で考えてよいが、ユーザー向けの説明、追加するルール、作業サマリーは日本語で書く。
- チャットに大きなコードブロックやファイル全文を貼らない。必要な変更はファイル編集として行う。

## 基本方針

- 実装コストよりも保守性、堅牢性、将来の読みやすさを優先する。
- 防御的な過剰設計よりも、既存コードに沿ったシンプルな解を優先する。
- 設計に入る前に、既存実装を検索して重複や競合作業を避ける。
- ユーザーの明示指示は字義通りに扱い、互換レイヤーやカテゴリ分割を勝手に追加しない。
- ユーザーが指定していない限り、未関係のリファクタやメタデータ更新を混ぜない。

## リポジトリ前提

- Laurel は Ghost テーマ互換の静的サイトジェネレーターであり、Markdown と Git 上のファイルを source of truth とする。
- 管理画面も file-first / Markdown-first の前提を壊さない。
- Ghost の巨大な CMS 機能をそのまま移植しない。静的サイトとして意味がある体験だけを選ぶ。
- Email / newsletter / members / paid tiers は Admin 改善の対象外とする。ただし既存ファイルに存在する未知 frontmatter は破壊せず保持する。

## Admin / Dashboard 改善ルール

- Ghost Admin / Ghost Editor は必ず研究対象にするが、模倣ではなく Laurel のファイル運用へ翻訳する。
- Ghost Koenig / Lexical の内部データモデルは原則移植しない。Markdown-first を維持する。
- Posts / Pages / Authors / Tags / Settings はファイルベースで扱う。
- Posts / Pages の一覧は作成日基準のページネーションを持つ。
- 外部エディタや別プロセスがファイルを変更する前提で、同期は双方向にする。
- 保存前 fingerprint と現在 fingerprint を比較し、外部変更を上書きしない。
- Preview は設定された active theme で、保存済み最新ファイルがどう見えるかを表示する。
- deploy しないと公開されない前提を守る。未保存内容の live preview を公開状態の代替にしない。
- 未保存変更がある場合は、Preview が保存済み状態に基づくことを明示する。
- ビルド済みブログ記事は Lighthouse 全カテゴリ 100 点を目標に、品質ゲートとしてテスト化する。

## デザイン方針

- Ghost より美しく使いやすい Admin を目指す。
- 業務ツールとして静かで密度の高い UI にする。装飾過多のランディングページ風にしない。
- カードの入れ子、過剰なグラデーション、単色に寄りすぎた配色を避ける。
- テキストや UI 要素がモバイル/デスクトップで重ならないことを必ず確認する。
- アイコンが適切な場面では lucide 等の既存アイコンを使う。
- フロントエンド文言は簡潔にし、可能なら `/ask-codex` で確認する。

## 実装前チェック

- 機能実装やリファクタリングの前に、潜在的なエッジケースと注意点を列挙する。
- 共有型、インターフェース、関数シグネチャを変える場合は、grep だけでなく LSP 相当の参照調査を行う。
- 既存の CLI、テスト、ドキュメント、スナップショットへの影響を確認する。

## ファイル変更時のルール

- ファイルを変更したら、次に進む前に必ずサマリーを提供する。
  - 変更したすべてのファイル
  - 各ファイルの主要な変更点
- ユーザーや別プロセスの未コミット変更を勝手に戻さない。
- 既存の dirty worktree では、自分の変更と無関係な差分を触らない。
- `git reset --hard` や `git checkout --` などの破壊的操作は、明示依頼なしに使わない。
- 手作業の編集は `apply_patch` を使う。

## PR ワークフロー

- 明示されない限り、PR は `main` ブランチに対して作成する。
- ローカルマージは禁止。常に `gh pr create` 経由の PR フローを使う。
- ただし `/worktree-parallel-flow` を使っている場合は、そのフロー内のローカルマージを許可する。
- `git commit --amend` は使わない。必要なら新しいコミットを作る。
- PR 作成後、依頼があれば worktree とローカルブランチをクリーンアップする。

## コミット除外ルール

- worktree 用のポート変更はコミットに含めない。
- `docker-compose.yml`、`.env.sample`、`provision/docker/docker-compose.yml` などのローカルポート変更は PR に含めない。

## Push 前チェック

- Push 前に `/run-github-actions-locally` を実行して CI 相当を確認する。
- 失敗する場合は修正して再度確認する。
- 成功した場合に Push する。
- 通常のローカル検証では少なくとも `bun run check` と `bun test` を実行する。

## レビュー

- 非自明な実装後は `/codex:review` でレビューを受け、問題があれば修正する。
- レビュー依頼時は、変更意図、影響範囲、通した検証を明記する。

## project-backlog

- Admin 改善の未着手タスク、設計判断、Ghost 研究メモは `project-backlog` に記録する。
- 別セッションや別ブランチで拾えるように、タスクには背景、参照ファイル、完了条件、除外範囲を含める。
- Email/newsletter は不要というスコープ決定を破らない。

## 並行作業フロー

- kiri 環境で別修正が発生した場合のみ、`kiri-cli` で別 pane を開いて並行作業する。
- 前提として `KIRI_TERMINAL=1` を確認する。
- それ以外の環境では本流を一時停止し、順次対応する。
